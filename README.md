# Adversarial Machine Learning for Network Security
A two-phase machine learning framework featuring an adversarial attack fuzzer and defensive hardening loop evaluated against the CICIDS2017 dataset.

## Project Overview

Intrusion Detection Systems (IDS) increasingly rely on machine learning to identify malicious network traffic. However, these models remain vulnerable to adversarial attacks deliberate, mathematical perturbations of network feature statistics designed to trick classifiers while preserving the underlying exploit's payload. 

This project bridges that gap by engineering a complete automated feedback loop that:
1. **Exposes Vulnerabilities:** Simulates an adversary using a feature fuzzer.
2. **Reinforces Defense:** Retrains and hardens the defensive models against the generated adversarial patterns.

---

## System Architecture

The framework operates in a continuous two-phase pipeline to systematically eliminate machine learning vulnerabilities:

Phase 1: Attack Fuzzer (Feature Perturbation) ──> Phase 2: Hardening Loop (Adversarial Retraining) ──> Repeat

### Phase 1: The Adversarial Attack Fuzzer
* Generates adversarial network traffic variations by executing targeted feature mutations on standard attack classes.
* Manipulates statistical packet variables (e.g., forward/backward packet lengths, flow durations, inter-arrival times) to successfully bypass standard machine learning classifiers.
* Ensures perturbations strictly adhere to network protocol constraints so the underlying malicious payload remains functionally viable.

### Phase 2: The Defensive Hardening Loop
* Captures the successful adversarial evasion samples generated by the fuzzer.
* Automatically labels and feeds these evasion vectors back into the primary training pipeline.
* Executes a retraining cycle, generating a hardened model with reinforced decision boundaries.

---

## Evaluation & Dataset

* **Dataset:** Evaluated against the **CICIDS2017 (Canadian Institute for Cybersecurity Intrusion Detection System)** dataset, focusing on real-world network traffic profiles including DDoS, Brute Force, and Infiltration patterns.
* **Core Metrics:** Performance is benchmarked using Accuracy, Precision, Recall, and F1-Score across standard, under-attack, and post-hardening operational states.

## Execution & Replication Guide

This guide provides the sequential operational steps required to replicate the research findings, moving from raw data ingestion to the verification of the hardened IDS architectures.

## Environment Setup
Before replicating the steps below, ensure your local virtual environment is active and dependencies are installed:
```bash
python3 -m venv .venv
source .venv/bin/activate  # On Windows: .venv\Scripts\activate
pip install -r requirements.txt
```

### Step 1: Data Sanitization
* **File:** `data_cleaner.py`
* **Action:** Execute the script to scrub the raw CICIDS2017 CSV files.
* **Purpose:** Removes NaN (null) values and Inf (infinite) flow markers that cause mathematical errors during Gradient Boosting and Random Forest training.

### Step 2: NBI Subset Generation
* **File:** `subset_generator.py`
* **Action:** Filters the cleaned data to isolate specific traffic classes.
* **Purpose:** Extracts a balanced subset of Normal (Benign), Botnet, and Infiltration traffic. This ensures the model focuses on the high-risk "stealth" classes targeted by the fuzzer.

### Step 3: Baseline Model Training
* **File:** `train_RandomForest.py` & `train_XGBoost.py`
* **Action:** Trains the initial Random Forest and XGBoost classifiers.
* **Purpose:** Generates the baseline `.pkl` model files. These models represent the "standard" IDS state before any adversarial exposure.

### Step 4: Victim API Initialization
* **File:** `victim_api.py`
* **Action:** Launches the Flask REST API to host the model, allowing you to select which model to use for testing.
* **Purpose:** Simulates a live network node. The API listens locally (`http://127.0.0.1:5000/predict`), waiting for traffic features to classify.

### Step 5: Initial Adversarial Attack
* **File:** `adversarial_generator.py`
* **Action:** Runs the triple-tap fuzzer against the active API.
* **User Input:** Select the desired intensity level and set the target number of attempts in the terminal prompt.
* **Purpose:** Identifies decision boundary gaps. Successful bypasses are automatically logged and saved to `successful_attacks.csv`.

### Step 6: Model Hardening
* **File:** `harden_ids.py`
* **Action:** Executes the retraining script using the harvested bypass data.
* **Purpose:** Merges the `successful_attacks.csv` back into the original training dataset with corrected labels. It retrains the models to recognize fuzzed variations as malicious.

### Step 7: Environment Hot-Swap
* **Action:** Terminate the current `victim_api.py` process, restart it, and select the newly generated hardened model.
* **Purpose:** Deploys the defensively aware model into the live simulation environment.

### Step 8: Post-Hardening Verification
* **File:** `adversarial_generator.py` & `evasion_tester.py`
* **Action:** Re-run the fuzzer against the hardened API. Use `evasion_tester.py` to evaluate performance when processing benign traffic.
* **Purpose:** Quantifies the reduction in ASR (Adversarial Success Rate). This final step confirms the effectiveness of the defensive hardening loop.
